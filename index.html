CASE 01 / 04

400여 제휴 골프장 정책을 상담 가능한 운영 데이터 구조로 표준화

예약 고객상담정책 데이터 표준화제휴사 확인지식 관리
내 역할 · 예약·취소·위약·동반자 변경 고객 문의 직접 응대 · 400여 곳 어드민 전수 점검 · 공통 속성·조회 순서 정의 · 누락 정책 확인 · 복합·예외 판단 기준 및 변경 관리

PROBLEM

카카오골프예약 고객센터에서 예약·취소·위약·동반자 변경 문의를 전화로 직접 응대했습니다. 고객의 취소 가능 시점이나 위약 적용 여부를 빠르게 확인해 안내해야 했지만, 400여 제휴 골프장의 정책 정보는 위치·항목·표현 방식이 제휴사마다 달랐습니다. 일부 골프장은 위약 적용 범위나 동반자 제한 여부가 비어 있거나 표현이 불명확해, 운영자가 매번 다른 순서로 정보를 찾고 해석해야 했습니다. 단순히 정보량이 많은 문제가 아니라, 동일한 문의도 담당자에 따라 검색 순서와 판단이 달라질 수 있는 정보 구조의 문제였습니다.

WHAT I DID

A
반복 확인 정보의 공통 속성 도출
400여 곳 제휴 골프장의 어드민 정보란을 전수 확인하고, 고객 문의를 처리할 때 반복적으로 확인하는 예약실 연락처, 운영시간, 취소 가능 기한, 위약 적용 범위, 위약금 입금 계좌, 동반자 제한 여부를 공통 속성으로 정의했습니다.
B
정보 위치·표현·조회 순서 표준화
제휴사별로 흩어져 있던 정보 위치와 문장 표현을 일정한 구조로 통일했습니다. 운영자가 필요한 내용을 위에서부터 같은 순서로 확인할 수 있도록 예약실 연락처 → 운영시간 → 취소 가능 기한 → 위약 적용 범위 → 위약금 입금 계좌 → 동반자 제한 여부 순서로 재정리하고, 같은 의미를 다른 문장으로 표현한 내용은 운영자가 빠르게 이해할 수 있는 문장 구조로 통일했습니다.
C
누락 정보 보완 및 지속 관리
정보가 누락됐거나 표현이 불명확한 경우에는 추정하지 않고 해당 제휴 골프장에 직접 확인해 보완 입력했습니다.

BEFORE → AFTER

Before

· 제휴사마다 정보 위치와 표현 방식이 다름
· 위약 적용 범위·동반자 제한 등 일부 항목 누락 또는 불명확
· 운영자별로 정보를 찾는 순서와 해석 방식이 달라질 수 있음
· 정책 변경과 예외 케이스가 개인 경험에 의존

After

· 예약실 연락처 → 운영시간 → 취소 가능 기한 → 위약 적용 범위 → 위약금 입금 계좌 → 동반자 제한 여부 순서로 통일
· 누락·불명확 내용은 제휴사 확인 후 보완
· 담당자가 달라도 동일한 순서와 기준으로 판단 가능

RESULT

제각각이던 비정형 정책 정보를 공통 속성과 일정한 조회 순서로 전환했습니다. 400여 제휴사의 정보를 단순 보관 데이터가 아니라 고객 문의를 처리할 때 바로 사용할 수 있는 운영 지식으로 구조화해, 담당자가 달라도 같은 순서로 정보를 확인하고 동일한 기준으로 예약·취소·위약 문의를 판단할 수 있도록 했습니다.

제각각이던 비정형 정책 정보
→ 공통 속성과 일정한 조회 순서를 갖춘 상담 운영 데이터

INSIGHT

비정형 정보는 단순히 모으는 것만으로 운영 기준이 되지 않습니다. 공통 속성, 조회 순서, 누락 확인, 예외 판단, 변경 관리까지 설계해야 담당자가 일관된 판단에 사용할 수 있는 운영 데이터가 됩니다. 제휴사마다 정책은 달라도 같은 순서와 기준으로 확인할 수 있게 만드는 것이 표준화의 핵심임을 배웠습니다.

트립톡 적용 · eSIM 상품은 국가·지역·통신사·지원 기기·데이터 용량·사용 기간·개통 방식·테더링·환불 조건이 상품마다 달라질 수 있습니다. 이 정보를 공통 항목과 일정한 조회 순서로 관리하고, 모호한 내용은 공급사·파트너 확인 후 온라인 마켓 상품 정보, FAQ, 상담 기준에 함께 반영하겠습니다. 고객 문의에서 반복되는 혼선은 개별 응대로 끝내지 않고 상품 정보 구조와 운영 지식 개선으로 연결하겠습니다.

CASE 02 / 04

조직 공백에서도 같은 기준으로 응대하도록 FAQ·검수·온보딩 체계 정비

FAQ·SOP답변 검수에스컬레이션온보딩
내 역할 · 문의 유형화 · FAQ·매크로·SOP 제작 · 답변 검수·피드백 · 처리 가능 범위·에스컬레이션 기준 정리 · 신규·지원 인력 온보딩

PROBLEM

조직 변동으로 기존 CS 인력이 순차적으로 이탈하고, CS 경험이 없는 타부서 지원 인력이 고객 문의를 처리해야 했습니다. 기존 문서와 실제 처리 기준이 여러 곳에 분산돼 있었고, 배송 지연·미출고·환불·교환·품질 이슈마다 확인해야 할 내용과 가능한 조치가 달라 개인 숙련도에 따라 답변과 처리 방향이 달라질 위험이 있었습니다.

WHAT I DID

A
문의 유형별 확인·처리 기준 정리
배송 지연, 미출고, 주문 취소, 환불·교환, 품질·민감 이슈별로 확인 항목, 고객 안내 범위, 가능한 조치와 후속 확인 절차를 FAQ·SOP로 정리했습니다.
B
답변 구조와 위험 표현 기준 표준화
답변을 공감 → 상황 요약 → 확인 결과 → 조치 → 최종 안내 순서로 정리하고, 확정·추정·확인 중 표현, 책임 주체, 사용하지 않아야 할 문구를 매크로와 검수 기준에 반영했습니다.
C
에스컬레이션과 답변 검수 체계 운영
CS에서 바로 해결할 수 있는 문의와 상품·물류·개발·경영진 판단이 필요한 문의를 분리했습니다. 지원 인력의 답변은 상황 요약 → 위험 포인트 → 권장 문구 → 다음 조치 기준으로 직접 검수하고 피드백했습니다.
D
온보딩 가이드 제작
신규·지원 인력이 문의 유형별 확인 순서와 이관 기준을 빠르게 익힐 수 있도록 FAQ, 처리 흐름, 예외 사례와 확인 주체를 하나의 온보딩 자료로 정리했습니다.

RESULT

· 평균 응대시간 3.1분 → 1.7분
· 매크로 사용률 10% → 62%
· 온보딩 기간 10일 → 4~5일
· 조직 변동 속에서도 고객 응대 공백 없이 동일한 처리 기준 유지

INSIGHT

좋은 FAQ는 답변 문구를 모아놓은 문서가 아니라, 어떤 정보를 확인하고 어디까지 해결하며 언제 다른 담당자에게 이관해야 하는지를 안내하는 운영 기준이어야 합니다. 문서 작성 이후 실제 답변 검수와 피드백까지 연결해야 기준이 현장에 정착된다는 점을 배웠습니다.

트립톡 적용 · 개통·설치·현지 연결·기기 호환·상품 조건·주문·환불 문의를 유형별로 정리하고, 주간·야간 담당자가 달라도 같은 확인 순서와 해결 단계로 안내하도록 FAQ·매크로·이관 기준을 관리하겠습니다.

CASE 03 / 04

상품 등록 직후 반복된 문의를 상품 정보·장바구니 안내 개선으로 해소

상품 정보 검수VOC 패턴안내 구조유사 상품 확장
내 역할 · 신규 상품 등록 직후 이상 문의 패턴 포착 · 주문·재고·화면 비교 · 고객 혼선 지점 정의 · 최소 안내안 제안 · 개발 협의 · 반영 후 문의 추이 확인

PROBLEM

화요일에만 도착 가능한 신규 상품을 등록한 직후, 기존에는 없던 동일 유형의 문의가 짧은 시간에 6~10건 발생했습니다. 배송 가능 요일이 다른 상품을 함께 담으면 주문이 차단됐지만 고객 화면에는 일반적인 품절형 메시지가 노출돼, 고객은 어떤 상품 때문에 주문이 불가능한지 알 수 없었습니다.

WHAT I DID

A
문의·주문·재고·화면을 비교해 원인 확인
신규 상품 등록 시점, 문의 발생 시점, 고객 장바구니 구성, 실제 재고 여부와 화면 문구를 비교했습니다. 문의마다 같은 배송 제약 상품이 포함돼 있었고 실제 품절이 아니어서, 개별 주문 문제가 아니라 상품 조건이 화면에 충분히 보이지 않는 공통 문제로 판단했습니다.
B
전체 주문 로직을 바꾸지 않는 최소 안내안 설계
차단 로직 전체를 변경하기보다 상품 옆에 배송 가능 요일과 나누어 주문이 필요할 수 있다는 안내를 노출하는 방식을 제안했습니다. 특정 상품에만 사용하는 문구가 아니라 상품별 조건에 맞게 적용할 수 있는 재사용형 subtext 구조로 정리했습니다.
C
당일 반영 후 유사 상품으로 확장
CTO·개발 담당자와 구현 범위를 확인해 당일 반영했습니다. 이후 토요일 도착 전용 상품이 추가됐을 때도 동일한 안내 구조를 적용해, 서로 다른 배송 가능 요일의 상품을 함께 담은 고객이 각 상품의 제약을 장바구니에서 확인할 수 있도록 했습니다.

RESULT

상품 등록 당일 6~10건 발생한 관련 문의는 반영 이후 확인 기간 내 0건 수준으로 감소했습니다. 고객이 상담을 요청하기 전에 상품 조건과 필요한 행동을 구매 화면에서 이해할 수 있게 했으며, 동일한 안내 구조는 다른 배송 제약 상품에도 확장 적용됐습니다.

반영 결과 화면
유사 배송 제약 상품으로 확장된 장바구니 안내 화면초기 화요일 도착 전용 상품에 적용한 안내 구조를 토요일 도착 전용 상품에도 확장해, 각 상품의 배송 가능 요일과 나누어 주문이 필요할 수 있음을 장바구니에서 확인하도록 했습니다.

INSIGHT

고객이 상품 조건을 이해하지 못한 문제를 단순 안내 부족이나 고객 실수로만 보면 같은 문의가 반복됩니다. 고객이 막히는 화면과 시점을 확인해 필요한 정보를 그 지점에 노출해야 상품 정보가 실제 구매 판단에 사용될 수 있습니다.

트립톡 적용 · eSIM 상품 등록 시 국가·사용 가능 지역·데이터 용량·사용 기간·통신망·개통 시작 조건·지원 기기·테더링·환불 제한을 구매 전에 이해할 수 있도록 검수하겠습니다. 동일 문의가 반복되면 상담 답변만 늘리지 않고 상품명·상세 설명·주의 문구·FAQ 중 고객이 막히는 지점을 찾아 개선하겠습니다.

CASE 04 / 04

고객 약속일을 기준으로 재배송·취소 1,928건의 처리 우선순위 재설계

주문 후속 처리재배송·취소고객 영향도시간 민감도
내 역할 · 처리 대상 분류 · 고객 영향도·시간 민감도 기준 우선순위 설정 · 반복 입력 자동화 · 진행 상태 확인 · 잔여 물량·지원 인력 재산정

SITUATION

금요일 3PL 전환 과정에서 재배송 1,151건과 취소 777건, 총 1,928건의 후속 처리가 발생했습니다. 재배송 대상 고객은 이미 한 차례 배송 불편을 겪은 뒤 도착일을 다시 안내받은 상태였습니다. 주말 동안 일반 주문이 늘면 고객에게 약속한 날짜의 주문 가능 물량이 마감될 수 있었지만, 취소 대상 777건은 재배송보다 시간 민감도가 낮아 월요일 일괄 처리가 가능했습니다.

SUCCESS GAP

접수 순서나 전체 물량만을 기준으로 처리하면 이미 불편을 겪은 고객의 재배송이 늦어져 다시 안내한 약속일까지 지키지 못할 위험이 있었습니다. 처음에는 전체 물량 기준으로 지원 인력 3명을 요청했지만, 재배송과 취소의 고객 영향도와 처리 시급성을 나눠 작업 순서와 필요한 인력을 다시 계산할 필요가 있었습니다.

WHAT I TRACKED

· 고객에게 다시 안내한 도착일
· 약속일 주문 가능 물량이 마감될 수 있는 시점
· 재배송 자동화 진행률
· 남은 취소 물량과 예상 처리시간
· 필요한 지원 인력 규모
· 전체 후속 처리 완료 여부

WHAT I DID

A
고객 영향도와 시간 민감도로 업무 분리
재배송 1,151건은 고객 약속일에 직접 영향을 주는 최우선 작업으로 지정하고, 취소 777건은 월요일 일괄 처리로 분리했습니다.
B
반복 입력 구간 자동화
재배송 처리의 반복 입력 구간에 Chrome Extension과 Apps Script를 적용해 1,151건을 주말 내 선행 완료했습니다.
C
잔여 물량 기준으로 지원 인력 재산정
재배송 완료 후 남은 취소 777건의 작업량과 처리 시간을 다시 계산해 지원 인력을 3명에서 2명으로 조정하고 월요일 오전 전체 처리를 마무리했습니다.

RESULT

고객에게 다시 안내한 약속일을 먼저 보호하면서 재배송 1,151건을 주말 내 완료했고, 취소 777건은 월요일 오전까지 처리해 전체 1,928건의 후속 조치를 마무리했습니다. 단순 접수 순서가 아니라 고객이 처한 상황과 시간이 지나면서 커지는 불편을 기준으로 업무 우선순위를 정한 경험입니다.

INSIGHT

모든 문의를 들어온 순서대로 처리하는 것이 항상 공정한 것은 아닙니다. 시간이 지나면 고객 피해가 커지는 문의와 이후 처리해도 영향이 상대적으로 적은 문의를 구분하고, 고객에게 이미 한 약속을 우선 보호해야 CX 관점의 우선순위가 됩니다.

트립톡 적용 · 출국 직전 eSIM이 개통되지 않거나 현지에서 연결되지 않는 고객은 일반 상품 문의나 단순 환불 요청보다 시간 민감도가 높을 수 있습니다. 문의 접수 순서만 따르지 않고 출국·체류 일정, 연결 상태, 이미 시도한 조치와 대체 가능 여부를 확인해 긴급도를 판단하겠습니다.

WORKING METHOD

고객 문의를 운영 개선으로 연결하는 방식

1. 상황 확인
고객의 현재 위치·시간 제약·주문 상태·상품 조건·이미 시도한 조치를 먼저 확인합니다.
2. 해결 순서 안내
고객이 바로 실행할 수 있는 항목부터 단계별로 안내하고, 확인이 필요한 경우 예상 회신 시점을 함께 전달합니다.
3. 예외·이관 판단
상담에서 해결 가능한 범위와 시스템·상품·파트너 확인이 필요한 범위를 분리합니다.
4. 반복 원인 개선
같은 문의가 반복되면 FAQ, 매크로, 상품 등록 정보, 화면 문구, 운영 기준 중 어디를 바꿔야 하는지 확인합니다.
AUTOMATION & AI

반복 업무를 줄이고 정확도를 높인 운영 도구

상담 자체를 자동화한 것이 아니라, 상담 이후 반복되는 입력·집계·접수·조회·검증·공유 업무를 줄여 고객 안내와 예외 판단에 더 집중할 수 있도록 구성했습니다.

반품·재배송 입력 자동화

Chrome Extension · Apps Script

재배송 등록을 6→2단계, 반품 등록을 7→1단계로 단축하고 귀책·사유 등 기준값이 자동 반영되도록 구성했습니다. 재배송 1,151건의 대량 처리와 이후 반품 등록 전 건, 상시 AS·재배송 업무에 적용했습니다.

운영 시트 자동화

Excel VBA · Google Sheets

중복·누락 확인, 미처리 건 식별, 상태별 표시와 반복 집계를 자동화해 처리 현황을 빠르게 확인하고 수기 입력 과정의 오류 가능성을 줄였습니다.

부서 간 요청 접수 자동화

Slack Workflow · Google Sheets

작성자·담당자·상품코드·상품명·요청 내용·이미지 등을 정형 항목으로 접수하고, 담당자가 태그된 Slack 메시지와 Google Sheets 처리 이력이 자동으로 남도록 구성했습니다.

상품 담당자 즉시 조회

Slack API · Apps Script

Google Sheets의 상품 담당자 데이터를 '/담당자' 명령어로 조회하도록 구성했습니다. 결과는 요청자에게만 표시하고, 담당자 변경은 Google Sheets 수정값을 기준으로 반영되도록 해 엑셀 파일을 반복해서 열어보던 과정을 줄였습니다.

반복 커뮤니케이션 자동화

Zapier · Slack
